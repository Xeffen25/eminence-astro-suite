The icons integration resolves icon metadata at build time from a file-keyed configuration. Each key is an output file name, and each value describes how that file should be generated, whether it should emit a `<link>` tag, and whether it should contribute an entry to `manifest.webmanifest`.

Icon resolution happens in three layers:

1. `icons`: generates assets and produces base head/manifest icon metadata from those keyed files.
2. `headTags.icons`: merged over generated head icon tags using `href` as the key.
3. `manifest.icons`: merged over generated manifest icon entries using `src` as the key.

When keys collide, the later layer overrides the earlier one.

During the build the integration:

- generates raster and `.ico` assets with `sharp` and `sharp-ico`
- resolves `href`, `sizes`, and `type` into static `IconTag[]`
- injects those tags through `virtual:eminence-astro-suite/head-tags`
- scans `manifest: true` entries to populate `manifest.webmanifest`

`sharp` and `sharp-ico` are required peer dependencies when you generate raster or `.ico` files.

## Options

### `IconsOptions`

| option                                         | type                                                                                                                                                                   | required | description                                                                                                                                                                                            |
| ---------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `source`                                       | `string`                                                                                                                                                               | Yes      | Global source image used for generated raster and `.ico` outputs unless a keyed entry provides its own `source`.                                                                                       |
| `[filename.ico]`                               | `{ sizes: number[] \| false; tag?: Partial<IconTag> & Pick<IconTag, "rel">; manifest?: boolean \| ManifestIconOptions; source?: string; sharpOptions?: SharpOptions }` | No       | Generates a multi-size `.ico` file. `sizes: false` skips the entry entirely. Link-tag `sizes` are never inferred for `.ico`; set them manually in `tag` if needed.                                     |
| `[filename.svg]`                               | `{ tag?: Partial<IconTag> & Pick<IconTag, "rel">; manifest?: boolean \| ManifestIconOptions }`                                                                         | No       | Metadata-only SVG entry. No file is generated for explicit SVG keys. If the global `source` is SVG and `favicon.svg` is not defined, the integration auto-copies the source and injects a favicon tag. |
| `[filename.<png\|jpg\|jpeg\|gif\|webp\|avif>]` | `{ size: number \| false; tag?: Partial<IconTag> & Pick<IconTag, "rel">; manifest?: boolean \| ManifestIconOptions; source?: string; sharpOptions?: SharpOptions }`    | No       | Generates a raster file at the configured square size. `size: false` skips the entry entirely. Raster link-tag sizes are inferred as `NxN`.                                                            |

### `IconTag`

| field   | type                          | required | description                                                                                                                                       |
| ------- | ----------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `rel`   | `string`                      | Yes      | Required to emit a `<link>` tag. If `tag.rel` is omitted, the file can still generate and contribute to the manifest, but no head tag is emitted. |
| `href`  | `string`                      | No       | Ignored in build config. The integration always derives `href` from the file key and normalizes it with a leading `/`.                            |
| `sizes` | `string`                      | No       | Optional explicit `sizes`. Used as-is for `.ico` or SVG metadata. Raster files infer this automatically from `size`.                              |
| `type`  | `string`                      | No       | Optional MIME override. Defaults to `inferImageMimeType(fileName)`.                                                                               |
| `media` | `"light" \| "dark" \| string` | No       | Optional media hint. The component renderer expands `light` and `dark` to `prefers-color-scheme` queries.                                         |

### `ManifestIconOptions`

| field     | type     | required | description                                                                                                                 |
| --------- | -------- | -------- | --------------------------------------------------------------------------------------------------------------------------- |
| `src`     | `string` | No       | Override the manifest icon path. Defaults to the normalized file key.                                                       |
| `sizes`   | `string` | No       | Override inferred sizes. Raster icons infer `NxN`; `.ico` icons infer a space-separated list such as `"16x16 32x32 48x48"`. |
| `type`    | `string` | No       | Override inferred MIME type.                                                                                                |
| `purpose` | `string` | No       | Optional manifest purpose such as `"maskable"` or `"monochrome"`.                                                           |

### `IconTag`

| field   | type                          | required | description                             |
| ------- | ----------------------------- | -------- | --------------------------------------- |
| `rel`   | `string`                      | Yes      | The `rel` attribute for the link tag.   |
| `href`  | `string`                      | Yes      | The `href` attribute for the link tag.  |
| `sizes` | `string`                      | No       | The `sizes` attribute for the link tag. |
| `type`  | `string`                      | No       | The `type` attribute for the link tag.  |
| `media` | `"light" \| "dark" \| string` | No       | The `media` attribute for the link tag. |

## Usage & Examples

### Basic usage

```ts
eminence({
	icons: {
		source: "src/assets/logo.svg",
		"favicon.ico": { sizes: [16, 32, 48], tag: { rel: "icon" } },
		"apple-touch-icon.png": { size: 180, tag: { rel: "apple-touch-icon" } },
		"icon-192x192.png": { size: 192, tag: { rel: "icon" }, manifest: true },
		"icon.png": { size: 512, tag: { rel: "icon" }, manifest: true },
	},
});
```

This generates the configured files, injects the resolved icon tags into the virtual module, and contributes the two manifest-enabled icons to `manifest.webmanifest` when the manifest integration is active.

If you also set `headTags.icons` or `manifest.icons`, those arrays are merged over the generated entries (by `href` and `src` respectively).

### Automatic SVG favicon

```ts
eminence({
	icons: {
		source: "src/assets/logo.svg",
		"favicon.ico": { sizes: [16, 32, 48], tag: { rel: "icon" } },
	},
});
```

Because the global source is SVG and `favicon.svg` is not explicitly configured, the integration automatically copies the source to `/favicon.svg` and injects:

```html
<link rel="icon" href="/favicon.svg" sizes="any" type="image/svg+xml" />
```

### Skipping specific outputs

```ts
eminence({
	icons: {
		source: "src/assets/logo.png",
		"favicon.ico": { sizes: false, tag: { rel: "icon" } },
		"icon-192x192.png": { size: false, tag: { rel: "icon" } },
		"apple-touch-icon.png": { size: 180, tag: { rel: "apple-touch-icon" } },
	},
});
```

Entries with `size: false` or `sizes: false` are skipped for generation, head-tag resolution, and manifest population.

### Adding manifest metadata per file

```ts
eminence({
	icons: {
		source: "src/assets/logo.svg",
		"icon-192x192.png": { size: 192, tag: { rel: "icon" }, manifest: true },
		"badge.png": {
			size: 96,
			tag: { rel: "icon" },
			manifest: { purpose: "monochrome", src: "/brand/badge.png" },
		},
	},
	manifest: {
		name: "My App",
		start_url: "/",
		display: "standalone",
	},
});
```

### Metadata-only SVG entries

```ts
eminence({
	icons: {
		source: "src/assets/logo.png",
		"safari-pinned-tab.svg": {
			tag: { rel: "mask-icon", color: "#111111" },
		},
	},
});
```

This emits a `<link rel="mask-icon">` tag for `/safari-pinned-tab.svg` but does not generate the file. The SVG must already exist in the final site output path.

### Explicit opt-out

```ts
eminence({
	icons: false,
});
```

No icons are generated and no warning is logged.

## Decisions Made

### File names are the API boundary

The config is keyed by output file name so the final public URLs are explicit and stable. This removes the old preset-name indirection and makes the manifest and head tags derive from the same source of truth.

### Build-time resolution keeps the renderer thin

The integration resolves `href`, `sizes`, and MIME type before the `Icons` component runs. That reduces runtime logic and avoids shipping repeated inference code into every page render.

### Manifest population scans the same keyed entries

Manifest icons now come from the same file entries used for generation and head tags. That prevents drift between generated files, rendered tags, and `manifest.webmanifest` icon metadata.

### `tag.rel` controls head-tag output explicitly

Generation and manifest population can happen without a rendered `<link>` tag. Requiring `tag.rel` makes head-tag output opt-in and avoids accidental link injection.

If `icons` is configured but `source` is not set, the integration logs a warning recommending you provide a source path or set `icons: false` to disable this feature explicitly.

### Setting `false` is the explicit opt-out

Use `icons: false` to disable icon generation and icon metadata resolution from this integration.

If you want to manage files manually but still control rendered icon tags and manifest icons, configure `headTags.icons` and/or `manifest.icons` directly.

### Generated icons opt in to manifest inclusion via metadata

Default generated icons do not automatically appear in `manifest.webmanifest`. Only `png192` and `png512` carry built-in manifest metadata, because those are the two sizes Chromium requires for a PWA to be installable. All other default icons (32px, 48px, `apple-touch-icon`) are excluded. Custom generated icons are only included when they explicitly set `manifest: { ... }` on their entry.

This design keeps the manifest icon list intentional and minimal by default while making it easy to extend.
